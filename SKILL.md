# Smart Excel 技能

本技能综合「建模可交付性」「自动化脚本效率」「原则与避坑」三类做法：**优先保证工作簿在 Excel 中逻辑正确、类型不被破坏、模板意图不被悄悄改写**；批处理场景用成熟库与可复用脚本模式，**不绑定单一厂商或单一重算工具**。

## 何时使用

- 主要产出或主要输入是 `.xlsx` / `.xlsm` / `.xls` / `.csv` / `.tsv`。
- 涉及公式、日期、单元格格式、多工作表、合并区域、条件格式或数据校验。
- 需要批量合并、筛选、拆分、去重、聚合、按模板填充或校验列规则。

**不必使用**：主交付物是 Word/HTML/纯 Python/数据库管道；或仅通过 Google Sheets API 云端协作且与本地文件无关。

---

## 1. 按任务选工具（避免「只会 pandas」或「只会手搓 openpyxl」）

| 任务类型 | 首选 | 说明 |
|----------|------|------|
| 探索、聚合、透视、过滤、多表对齐 | **pandas**（`engine="openpyxl"`） | 不写回时也可用 Polars 等，最终落 xlsx 再交 openpyxl |
| 保留/修改公式、样式、批注、合并格、列宽、数据验证 | **openpyxl**（或按团队约定：xlsxwriter 仅写、xlwings 若本机 Excel） | 大文件只读用 `read_only=True`，只写用 `write_only=True` |
| 旧版 `.xls` | **xlrd**（只读）等 | 写入需另存为 xlsx 或专用库，先确认许可与兼容性 |
| 公式需要「算出缓存值」再交付 | 见下文「重算策略」 | openpyxl **不会**计算公式 |

---

## 2. 公式与工作簿「可编辑性」（避开「Python 算完写死数」）

- **若用户需要可刷新、可审计的表格**：把合计、比率、同比等写成 **单元格公式**，引用假设单元格，避免把 Python 计算结果硬写入结果格（除非用户明确要求静态快照）。
- **交付前**：工作簿应 **无** 未处理的 `#REF!`、`#DIV/0!`、`#VALUE!`、`#NAME?`；对模型类任务尽量避免无意循环引用。
- **复制公式**：注意相对/绝对引用；大范围填充前先抽 2～3 格手算或对照验证。
- **读模式陷阱**：`load_workbook(..., data_only=True)` 读到的是**上次保存的缓存值**；若再 **save**，可能把公式覆盖成数值——仅在明确要「扁平化」时使用，且需用户知情。

---

## 3. 重算策略（不强制单一依赖，按环境选）

openpyxl 写入的公式字符串不会自动变成最新计算结果。按环境择一或组合：

1. **用户用 Microsoft Excel 打开并保存**：最贴近最终使用者；适合小团队交付说明。
2. **本机已装 LibreOffice**：可用 `soffice` 无界面转换/重算类流程（若你项目中有封装脚本则优先调用项目脚本），注意首跑配置与超时。
3. **自动化 CI**：优先在说明中要求「验收方用 Excel 打开保存」或提供 Docker/镜像中的 LibreOffice headless（由项目维护，本技能不绑定具体路径）。

**最低限度**：若无法自动重算，须在交付说明中写明「请用 Excel 打开后按 F9/保存以更新」，并仍完成 **公式错误扫描**（可用 openpyxl 遍历 `data_only=False` 检查单元格是否以 `#` 错误开头，或对已知区域抽样）。

---

## 4. 日期、时间与类型（避开「看起来对、其实已经坏」）

- Excel 日期多为 **序列数** + 显示格式；存在 **1900 闰年 bug**、**1904 日期系统** 等历史问题，跨工具打开可能漂移。
- **长数字、身份证、单号、带前导零**：写入前设为 **文本**（列格式 `@` 或前置 `'`），避免科学计数法与 15 位精度截断。
- **读入时**：对关键列显式 `dtype` / `converters`，避免整列被误解析成日期或浮点。
- **混合列**：宁可在 pandas 中先 `astype(str)` 再按需清洗，也不要依赖默认推断。

---

## 5. 模板与结构保留（避开「偷偷换了一套样式」）

- **已有模板**：新填区域对齐现有字体、边框、数字格式；不强行套用另一套「财务配色规范」除非用户要求。
- **合并单元格**：值通常在 **左上角**；改表时注意合并范围，避免写入被合并覆盖的「幽灵格」。
- **隐藏行列、名称、外部引用、数据验证、条件格式**：可能承载业务规则；修改前确认是否要保持。
- **打印区域、冻结窗格、筛选**：若用户关心版式或打印，修改后复核。

---

## 6. 规模与性能

- **大表**：避免一次性 `read_excel` 无界膨胀；指定 `usecols`、`nrows`、sheet，或分块；写用 `write_only` 或分块追加。
- **空行 padding**：部分文件底部大量空行会导致慢与内存尖峰，读取时可设边界或 drop 全空行。

---

## 7. 常见任务速查（可直接用库完成，不依赖特定仓库脚本名）

| 目标 | 推荐做法 |
|------|----------|
| 多文件多 sheet 合并成一张表 | pandas `concat`，注意表头与重复列名 |
| CSV ↔ xlsx | `read_csv` / `to_excel` 或 openpyxl 逐行（超大用 write_only） |
| 按条件筛行 | `df.query` / 布尔索引，再 `to_excel` |
| 按列去重 | `drop_duplicates` |
| 分组聚合 | `groupby.agg` |
| 两表按键对齐 | `merge`（`how` 选 left/inner/outer） |
| 转置 | `df.T` 或 `pivot`/`melt` 视语义 |
| 按行套模板 | 保留模板 sheet，openpyxl 复制行或填占位符（如 `{{列名}}`），注意合并格 |
| 列强制文本 | 设置 `number_format='@'` 或写入字符串 |

若团队目录中有 **专用 CLI 脚本**（如 `merge_sheets.py`、`validate_excel.py`），优先复用脚本并保持参数与日志一致，而不是每次重写。

---

## 8. 批量处理与健壮性

- 遍历目录时用 `pathlib`；单文件失败可记录并 **继续**（按用户要求）。
- 覆盖已有文件前确认；大文件写入失败时避免留下半截文件（可先写临时文件再 `replace`）。
- 捕获 `InvalidFileException`、工作表 `KeyError`、编码错误，返回 **文件名 + 可读原因**。

---

## 9. 交付检查清单（精简）

- [ ] 公式无常见错误类型；关键引用已 spot-check。
- [ ] 日期/长数字列显示与类型符合业务预期。
- [ ] 模板类文件：样式、合并、验证、条件格式未意外破坏。
- [ ] 若含公式：已说明重算方式，或已在本环境完成可验证的重算。
- [ ] 大文件：读写策略与内存风险已考虑。

---

## 10. 生成代码时的风格

- Python **简洁**：少无意义 print；变量名清晰即可，避免冗长注释。
- **工作簿内**：复杂假设或硬编码可在批注/旁列说明来源与日期（用户或合规要求时）。

---

## 与其他技能的关系（概念上）

- 需要 **纯文本表格** 互转：配合 `csv` 类流程，再导入 xlsx。
- 需要 **统计分析后再出表**：分析在 pandas/R 等完成，**定稿规则**仍遵守本文第 2～5 节。

## 版本说明

`smart-excel` 旨在：**比单一「财务建模 skill」更少强制版式**、**比纯脚本 skill 更强调公式与类型安全**、**比纯原则 skill 更可执行（表格与检查项）**；重算路径保持 **可选与可说明**，避免锁死单一外部程序。若项目另有 `LICENSE` 或内部规范，以项目为准。
